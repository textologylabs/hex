## [Unreleased]

### Fixed

- M15.7 — self-update safety for enterprise / air-gapped environments. `hex` checks npm for a newer version and may `npm i -g` on interactive launch; in a locked-down bank env that path needed to be provably non-blocking and centrally disable-able. **Verified + tested the fail-safe guarantees:** the check is skipped entirely unless **both** stdin and stdout are TTYs (so CI, pipes, and headless/automation never trigger a network call), the registry fetch is bounded by a 2-second timeout, and **every** failure mode — offline, proxy hang, non-2xx, malformed body, abort — is swallowed to `null` so a blocked check never delays beyond the timeout or errors the command you ran. The decision logic is refactored into pure, exported helpers (`shouldCheck`, `fetchLatestVersion` with an injectable `fetch`, `compareVersions`, `isNewerVersion`) — `src/update.ts` previously had **no tests**; it now has 12 (env opt-out, config opt-out, both non-TTY cases, version ordering, and the five fetch fail-safe paths). **New central opt-out:** a `update:` block in `~/.hex/config.yaml` — `update: { check: false }` disables the check via shared config, so a platform team ships the opt-out once instead of relying on every shell exporting `HEX_NO_UPDATE_CHECK=1` (config-load failure is swallowed — a broken config never blocks startup). **Documented** the full network posture + both opt-outs in the README (new "Self-update" section) and `docs/security.md`. Suite: 929 / 929.

- M15.5 — cancel + crash UX at the CLI boundary. Two rough edges from the cold-user pass. **(1) Cancel rendered as an error.** Esc-ing (or ctrl-C-ing) out of a `hex new` prompt threw `PromptCancelledError`, which hit the top-level catch and printed `error: prompt cancelled` then exited 1 — a deliberate cancel dressed up as a failure. The top-level handler now special-cases it: a quiet dim `cancelled` and exit **130** (the conventional "interrupted by user" code, so a script can tell cancel apart from both success and a real error). The mapping is extracted to a pure `describeTopLevelError` (`src/cli-errors.ts`) — `cli.ts` runs `main()` at import time, so the logic lives in a separately testable module. **(2) Mid-render crash left a partial dir with no hint.** Unlike `deploy`/`setup`/`upgrade`, `hex new`'s render/hook phase had no try/catch, so a failure (e.g. a throwing `post_render` JS hook) propagated raw. The render-onward phase is now wrapped: the spinner stops with `render failed`, the error prints as a friendly one-liner (`hex new failed during render: …`), and the outro names the **partial output directory** and points at `hex doctor` for recovery — exit 1, no raw stack. A genuine cancel inside the wrapped phase (Esc during the setup loop) is re-thrown so it still reaches the clean cancel path. Verified end-to-end against the real CLI (a hook-explosion template → friendly message + exit 1; happy path still exit 0). 3 new tests (`describeTopLevelError`: cancel → 130, Error → 1, non-Error throw → 1). Suite: 917 / 917.

### Added

- M15.17 — non-interactive `hex new --answers <file>`. The M15.15 cold-user pass found that `hex new` was interactive-only — enum/select prompts can't be driven headless, so CI, scripted, and reproducible scaffolds were impossible. **New `--answers <file.yaml>`** renders without prompts: the file is a YAML mapping of prompt name → value (recipe-child answers nest under the slot key, e.g. `api: { port: 8080 }`), every omitted prompt falls back to its default, and supplied values are validated against the manifest (type, `pattern`, `min`/`max`, enum membership) reusing the same rules the interactive widgets enforce. A required prompt with no default + no supplied value, a bad value, or an unreadable file fails with a clear, prompt-named message and a **non-zero exit** — it never hangs waiting for input, so a pipeline can gate on it. Implemented as a single seam: `runPrompts` gained an optional `supplied` map (and a `coerceSuppliedAnswer` validator + `defaultFor` fallback) that every prompt path already funnels through, so components, recipe-level, and recipe-child prompts are all covered; `runRecipePrompts` threads the namespaced sub-map per child (a missing child namespace resolves via defaults rather than falling back to a prompt). A `createNonInteractivePrompter` guards against a stray widget call, and `--answers` implies a non-interactive post-render (setup tasks left pending for `hex setup`) and requires an explicit template argument. **Verified end-to-end against the real CLI**: `vite-ts-spa` renders headless with the enum `license` applied + the rename hook fired; omitted prompts default; a bad enum, a missing required answer, and a missing file each abort with exit 1. 18 new tests (answers-mode engine resolution + per-type coercion/validation + when-skip; recipe namespacing + child defaults + required-throw; answers-file parsing). Docs updated — getting-started gains a "Scaffold non-interactively (CI / scripts)" section, the CLI reference documents the flag, and the M15.15 "no headless mode yet" caveats are lifted. - M15.3 — `run:` setup-task trust model + `docs/security.md`. The setup-task executor's only gate was a **hardcoded** allowlist (`npm`, `npx`, `node`, `git`, …). But an allowlisted binary is a launchpad — `npx <anything>`, `npm install` (install scripts), `node -e`, `git` hooks all reach arbitrary code — so for a remote (git/catalogue) template the allowlist was guard-rail, not sandbox, and there was no per-org or per-source control. A bank security review would stop at the door. M15.3 makes trust explicit, configurable, and documented. **Configurable allowlist** — a new `trust:` block in `~/.hex/config.yaml`: `trust.allowlist` overrides the built-in safe list (an empty list blocks *every* `run:` task — scaffold-files-only mode), and `validateSetupTasksAllowlist` takes the resolved list. **Source-trust gating** — the allowlist says *which binaries*; trust says *whose commands*. `file:` sources stay trusted to auto-run (you chose the path); a remote source auto-runs its `run:` tasks only when its identifier is in `trust.sources`. A new `src/core/setup/trust.ts` (`resolveTrustPolicy`, `isSourceTrusted`, `classifySource`) holds the pure decision. **Untrusted-remote UX** — when scaffolding from an untrusted remote source in a TTY, `hex new` no longer runs anything silently; it asks: **Trust** (persist to `trust.sources`, run, never ask again for it), **Review each** (run the auto-pass with a y/N confirm on *every* command — a new `requireConfirm` flag threaded through `runExecutorPass` → `runRitualTask`), or **Skip** (leave pending for `hex setup`). In a non-interactive context (CI / piped / `--no-setup`) an untrusted remote source's tasks never auto-run — they're left pending; pre-trust the source in config to auto-run headless, which keeps trust an auditable artifact. **`hex hive add --trust <url>`** vouches for a source in one step (and `trustSource` in `src/core/config/write.ts` persists it, document-preserving + idempotent, like the M15.1 writers). **`docs/security.md`** is the new consumer-facing threat model: the QuickJS-WASM hook sandbox (no fs/process/network, 5 s CPU / 32 MiB caps, `--trust-local` gradient), the `run:` allowlist + source-trust gates, marketplace signature verification (fail-closed), and the self-update network call (`HEX_NO_UPDATE_CHECK`). 18 new tests (trust policy + classify + configurable-allowlist enforcement, `trustSource` round-trips, `requireConfirm` ritual behaviour). - M15.16 — release-prep workflow (version bump + CHANGELOG consolidation). `release.yml` automated the *publish* half (tag → check + build + `npm publish` + GitHub Release), but the *prep* half was fully manual — bump `package.json`, move the `## [Unreleased]` CHANGELOG entries under a dated `## [X.Y.Z]` heading, commit, PR — exactly the hand-work M14.6 did for v0.9.0. **New `.github/workflows/release-prep.yml`** (`workflow_dispatch`, run from the Actions tab) takes a `version` input (an explicit `1.2.3` or `major`/`minor`/`patch`), runs `npm version --no-git-tag-version` to bump `package.json` + lockfile, consolidates the CHANGELOG via the new `scripts/release-changelog.mjs`, sanity-builds, then commits to a `release/vX.Y.Z` branch and opens a release-prep PR. **It stops at the PR by design** — the maintainer reviews + merges, then pushes the `vX.Y.Z` tag, which is what triggers `release.yml` to publish (and that workflow's tag-matches-`package.json` guard is the backstop). `scripts/release-changelog.mjs` (`releaseChangelog(text, version, date)`) renames the current `## [Unreleased]` section to `## [version] — date` and re-seeds a fresh empty `## [Unreleased]` on top (Keep a Changelog convention), trimming surrounding blank lines; it throws if Unreleased is empty so a release can't be cut with no body. Author identity is set via `GIT_AUTHOR_*` env on the commit step (the repo's convention) rather than mutating git config. 4 new tests cover the changelog transform (move + re-seed, empty-Unreleased throw, missing-section throw, no duplicate blank lines). Suite: 878 / 878. Part of the M15 — 1.0 go-live epic.
- M15.2 — CI workflow (`npm run check` on every PR + push). Until now the only GitHub Actions workflow was the tag-triggered `release.yml`, so Hex's own 874-test suite ran *only* at release time (gated by `prepublishOnly`). Between releases, `main` was unverified — a PR that broke a test or lint landed silently until someone happened to cut a tag. For a bank engineering review that's a non-starter; automated testing gating every change is table stakes. **New `.github/workflows/ci.yml`** runs on every `pull_request` and on `push` to `main` (the split avoids double-running a branch that has an open PR): `actions/checkout@v4` → `actions/setup-node@v4` (with npm cache) → `npm ci` → `npm run check` (typecheck + lint + test), across a **Node 20 + 22** matrix (`fail-fast: false` so one version's failure doesn't mask the other's). A `concurrency` group cancels superseded in-progress runs on the same ref so a rapid push sequence doesn't pile up jobs. The workflow guards the Hex repo itself — distinct from the deploy workflow the M12.4 `cicd-github-actions` provider emits into *generated* apps. No source touched; one workflow file. Second ticket of the M15 — 1.0 go-live epic (the first of two go-live blockers).
- M15.1 — `hex hive`, the honeycomb hub. The discovery + source-management commands were scattered across four top-level nouns — `hex sources` (list/refresh configured roots), `hex search`, `hex browse`, and `hex marketplace` (a lone `validate` lint) — so `hex marketplace` read as a hollow storefront while the verbs you'd expect under it lived elsewhere, and adding a catalogue meant hand-editing `~/.hex/config.yaml`. M15.1 consolidates all of it under one umbrella noun, **`hex hive`** (the honeycomb hub where the hex cells are stored — `marketplace` implies commerce, `catalogue` is a mouthful with a catalogue/catalog spelling trap). **Umbrella** (`src/commands/hive.ts`): `hex hive list` (the default action — bare `hex hive` lists configured sources + catalogues with `namespace · N packages`), `hive refresh`, `hive search <query>`, `hive browse [category]`, `hive validate [path]` are the existing surfaces re-homed; each command file now exposes a `build<X>Command(parent)` helper so the legacy top-level command and the hive subcommand share one definition. **Two net-new verbs**: `hex hive add <url>` / `hive remove <url>` read-modify-write `config.yaml` so onboarding never touches YAML by hand — `add` defaults to a `catalogue:` source (`--git` / `--path` pick the other kinds, `--ref` pins a git ref), is idempotent (an identical entry is a no-op), and writes block-style YAML via the `yaml` package's document API so existing comments + other top-level keys survive the round-trip (atomic write through `writeFileAtomic`); `remove` drops every source matching the given URL/path regardless of kind/ref. `hex hive info <pkg>` surfaces `Catalogue.listVersions` — the per-package view no command exposed before — listing a package's published versions (newest first) and where it resolves from, across every configured marketplace and catalogue; a qualified `acme/api` restricts the lookup to that namespace. **Backwards compatibility**: `sources` / `search` / `browse` / `marketplace` stay registered as **hidden aliases** (via commander's `{ hidden: true }`) so muscle memory and any catalogue-repo CI calling `hex marketplace validate` keep working — they're just gone from `--help`. `hex new` and `hex list` deliberately stay top-level (scaffolding verbs, not source-discovery ones). **New** `src/core/config/write.ts` (`addSource` / `removeSource`, document-preserving). 16 new tests: config-writer round-trips (fresh-file creation + block style, idempotent re-add, ref-distinguishes-entries, git/path shapes, comment + other-key preservation, remove-by-identifier, remove-nothing), `collectPackageInfo` / `formatPackageInfo` against a `file://` fake registry (versions newest-first, qualified-namespace filter, not-found), and a registration contract test (hive visible + the four aliases hidden-but-registered + all eight subcommands grouped under hive). Suite: 874 / 874. First ticket of the M15 — 1.0 go-live epic.

## [0.9.0] — 2026-06-11

First published release of `@hexology/hex` on npm. v0.9.0 closes the M14 release pass and lands the work that turns Hex from a working local scaffolder into a tool a teammate can install and use end-to-end without a clone.

**Phase 1 — configurable scaffolder.** M8 added stubbable components (in-process + out-of-process engines, recipe-side dedup, fixtures, prod-clean lint). M9 built the marketplace foundations (signed `hexpkg/1` format, `MarketplaceSource`, qualified `<marketplace>/<name>@<version>` addressing, multi-marketplace aggregation, block/override policy, `hex search` / `hex browse`, and a reference registry server). M10 made generated apps self-describing with a `.hex/lockfile.yaml` (schema + writer + reader + integrity verification + `hex doctor` integration). M11 shipped the upgrade engine (pristine reconstruction from the lockfile, stepwise chain walk, declarative + JS migration vocabulary, 3-way merge with git-style conflict markers, `hex upgrade` + `--continue` + `--abort`, user-tree escape hatch, modify/delete orphan handling).

**Phase 2 — deploy + CI/CD (M12).** First-class `DeployAdapter` + `CicdProvider` interfaces, a `hex deploy` command driven by the lockfile, a Vercel deploy adapter, a `cicd-github-actions` provider that emits a `.github/workflows/deploy.yml`, the `vite-ts-spa` reference template that dogfoods the whole stack, and the user-facing docs.

**M13 — git-catalogue marketplace.** The middle-ground path between local sources and a hosted registry: a `marketplace.yaml` repo lists packages + per-version git coordinates + block/override policy. `catalogue:` is a new `~/.hex/config.yaml` source kind alongside `path:` and `git:`; `hex list` / `hex search` / `hex browse` walk catalogues; `hex new <ns>/<name>@<spec>` resolves through them. A starter `templates/marketplace-catalogue` and `hex marketplace validate` complete the publish loop.

**M14 — release pass.** The work in this release that makes v0.9.0 shippable: package preparation + MIT LICENSE alignment (M14.1), tag-triggered npm publish + GitHub release workflow (M14.2), the `§4 walkthrough` matrix that surfaced the remaining footguns (M14.3), the three-action setup task model (`run:` / `open:` / `detail:`) with auto-execute pass and per-task hand-off ritual (M14.7 + M14.11), `hex doctor` surfaces outstanding setup tasks + `--json` (M14.8), `hex search` / `hex browse` walk `path:` / `git:` source roots so local-template dev loops work (M14.9), and Vercel CLI via `npx` so no global install is required (M14.10).

### Added

- M14.11 — Setup-task hand-off ritual (narrate → confirm → open → pause → run). M14.7's executor fired `open:` and `run:` actions back-to-back: the browser popped up suddenly, the task was immediately marked done, and the next task started before the user had even minted whatever they were supposed to mint. Surfaced as a real footgun in the §4 walkthrough — Captain hit it during the Vercel-token mint → `gh secret set` sequence. The fix is a structured hand-off model that announces what's about to happen, gets the user's consent before destructive/disorienting actions, performs the side effect, then pauses for the user to come back. **New ritual module** (`src/core/setup/ritual.ts`). `formatPlan(task)` derives a structured narration body from `run:` / `open:` / `detail:` in their natural sequence — `1. Hex will open <url> 2. <detail prose> 3. Hex will run: <cmd>` with a trailing "Control returns to this terminal after the browser" note for `open:`-bearing tasks. `runRitualTask(task, ctx)` orchestrates the full sequence with two effects bags injected: `RitualEffects` (`narrate`, `confirm`, `awaitContinue` — the human-facing prompts) and the existing `RunEffects` (`spawn`, `openUrl` — mechanical side effects from M14.7). For `open:`-bearing tasks: narrate → confirm → openUrl → awaitContinue → spawn(if any). For `run:`-only tasks: narrate → spawn (the global "Run N tasks now?" already collected consent). For `detail:`-only tasks: narrate only. **New `declined` outcome** (`RitualOutcome` extends `TaskRunOutcome`). When the user says N to confirm, the task stays pending and the orchestrator records `kind: 'declined'`; `ritualOutcomeSucceeded` returns false. **Executor pass rewires** (`src/core/setup/executor-pass.ts`). `runExecutorPass` accepts an optional `ritualEffects` (test seam) and routes every actionable task through `runRitualTask` instead of `runSetupTask`. ### Fixed

- Action-handler coverage thin — the commander `.action()` callbacks for `hex new` (post-render TTY-vs-`--no-setup` decision) and `hex setup` (no-checklist-found branch with its stderr message + `process.exitCode = 1`) were only covered by indirection. Fixed by mirroring the M3.7 pattern: pull the branch logic out of the callback into pure / dependency-injected functions. `src/commands/new.ts` gains a pure `planPostRender(result, { isTTY, setup })` returning `{ kind: 'no-tasks' } | { kind: 'has-tasks', interactive, initial, pendingCount, setupMessage? }` — the `isTTY && opts.setup` decision is now one expression in a function, not a 2-line `if` in the spinner block. `src/commands/setup.ts` gains a `runSetupCommand(cwd, effects)` taking a `SetupCommandEffects` (stderr / setExitCode / printIntro / printOutro / prompterFactory), plus a `defaultSetupCommandEffects` the production registrar wires; tests inject capturing effects to assert the no-checklist branch sets exit code 1 and writes the user-facing message to stderr without ever touching `process`. 7 new tests — 5 for `planPostRender` (no-tasks, every isTTY × setup combination, setupMessage propagation) and 2 for `runSetupCommand` (the no-checklist branch + the happy-path session drive).
- Checklist concurrency — two `hex setup` invocations against the same generated app's `.hex/checklist.yaml` (or one running while `hex new` finalises) could clobber each other's toggles via last-writer-wins on the whole file. The setup loop now passes the *specific* toggle (`{ taskId, status }`) to its `onSave` callback, and a new `updateChecklist(rootDir, mutate)` helper applies the mutator inside the same `withFileLock` primitive M3 cache concurrency added: it serialises read-modify-write, re-reads the latest disk state inside the lock, and writes back atomically — so two terminals each toggling a different task end up with *both* toggles on disk, not one. `writeChecklist` keeps its existing role (full overwrite, e.g. the initial write by `hex new`) but now writes atomically via `writeFileAtomic`, so a concurrent reader never sees a half-written file. `hex setup`'s persist path writes through `writeChecklist` if no file exists yet (first-toggle creation) and through `updateChecklist` otherwise. 4 unit tests cover the new helper (mutator applied + reloads clean, throws when no checklist, rejects a mutator that produces a malformed shape, and a 5-worker concurrent-toggle race where every toggle lands).
- Cache concurrency — two `hex` invocations against the same `(url, ref)` could race during clone/refresh: the meta file write (`.hex-meta.json`) wasn't atomic, and the whole read-then-mutate flow in `resolveGitSource` had no mutual exclusion, so two cold resolves could both try to `git init` / `git fetch` into the same repo dir and a `checkUpstreamDrift` could stomp the freshly-written meta. A new `src/core/util/atomic.ts` ships two dependency-free primitives — `writeFileAtomic` (write-to-temp + `rename`, so a reader observes either previous or new bytes, never a half-written file) and `withFileLock` (exclusive-create lockfile with `staleMs` reclaim, so a crashed process doesn't wedge things). `writeMeta` becomes atomic; `resolveGitSource` keeps the warm-cache fast path lock-free (safe now that reads are consistent) and serialises the clone/refetch under a `.lock` sentinel in the cache root, re-reading meta inside the lock so a peer-completed refresh short-circuits; `checkUpstreamDrift` runs `ls-remote` outside the lock (it's a slow network call) and persists the result under the lock so a concurrent resolve can't lose `lastCheckedAt` / `lastKnownUpstreamSha`. 8 tests cover the primitives (atomic write, lock serialisation, release on success/throw, stale reclaim, timeout, parent-dir requirement) and 2 cover end-to-end concurrency (5 cold resolves and 5 warm-cache refreshes against the same upstream — all converge on the same sha with a valid meta and no leftover lock).

- M10.1 — Lockfile schema. The first M10 ticket defines `.hex/lockfile.yaml`, the file that makes a generated app *self-describing* — the inverse of an authored component's `.hex/manifest.yaml` (manifest: *how to scaffold*; lockfile: *what was scaffolded*). The new `core/lockfile/` module ships a zod schema for the document: a `root` artifact (the recipe — or, with no `children`, a standalone component, so the engine keeps its no-`if (recipe)` uniformity), a `children` array (each with its `composes:` slot key + stub flag), the full `answers` tree, and a per-file `files` table of lowercase-hex sha256 digests over the rendered tree. Each artifact carries a discriminated `source` spec — `file` / `git` / `marketplace`, one variant per `Source` implementation — so M11 can re-fetch the exact bytes for pristine reconstruction. `schema_version` is `1`; the base schema accepts any positive integer there on purpose, leaving a future-version file for the M10.3 reader to refuse with an upgrade hint rather than a bare schema error. sha256 reuses the marketplace package format's hashing primitive — one content-hash algorithm across the codebase. `docs/lockfile-format.md` is the format spec; 9 tests cover the shape, every source-spec variant, and tampered/malformed-hash rejection. Writing the file is M10.2; reading + integrity verification is M10.3.
- M9.9 — Public Hex marketplace MVP (registry server). A new top-level `registry/` holds the marketplace registry server — a single dependency-free Node `http` service that closes the M9 epic by hosting the package format the rest of the epic was built against. **Read side**: `GET /catalogue.json`, `GET /<name>/index.json`, `GET /packages/<file>.hexpkg` — the exact static-shaped endpoints `MarketplaceSource` (M9.2) and the `Catalogue` (M9.3) fetch. **Publish**: `POST /publish` accepts a bearer-token-authenticated upload of an *unsigned* bundle tarball, validates it, **signs it with the marketplace key**, and ingests it — developers hold a publish token, not a signing key (marketplace-as-signer, M9.1); publishes are serialised so catalogue/index regeneration never interleaves, and an existing `name@version` is refused (immutable). **Website**: server-rendered search / browse / per-package detail pages, progressively enhanced with HTMX (the `htmx.org` dependency, served from `/assets/htmx.min.js` — never a third-party CDN) for live search. A new `hex publish <dir>` CLI command packs a bundle and uploads it; `registry/keygen.ts` (`npm run registry:keygen`) generates the marketplace keypair. `docs/registry-deploy.md` is the deploy runbook (keypair, tokens, env, TLS, smoke test). 13 tests cover the store, publish flow, every endpoint, and an in-process publish→fetch loop that resolves a just-published package back through `MarketplaceSource` with signature verification. DNS / TLS / hosting on Textology infrastructure remain for the operator.
- M9.8 — `hex browse` command. The sibling discovery surface to `hex search`: `hex browse` lists the marketplace-curated categories across every configured marketplace (a `name (count)` tally folded from the aggregate `Catalogue`), and on a TTY presents an interactive picker that drills down into the chosen category's entries. `hex browse <category>` lists a category directly, skipping the picker; piped (non-TTY) output prints the flat category list. `--json` emits `{ categories, warnings }` (or `{ category, results, warnings }` for a category argument). Block policy is honoured — blocked entries never reach the category tally or the drill-down. Drill-down entries reuse `hex search`'s table formatter. The testable core is `listCategories` + `browseCategory` + `formatCategories`; 6 tests run against a `file://` fake-registry fixture.
- M9.7 — `hex search` command. The first CLI surface for the marketplace: `hex search <query>` walks every marketplace in `~/.hex/config.yaml`, aggregates the hits via the M9.5 aggregate `Catalogue`, and prints aligned `qualified-name@version  type  description` rows. Block/override policy is loaded and applied — blocked entries are dropped from results. `--json` emits `{ results, warnings }` for scripting (mirroring `hex list` / `hex sources`); unreachable marketplaces and policy-load failures surface as warnings rather than aborting. With no marketplaces configured the command prints a config hint. The core is the testable `searchMarketplaces` + `formatSearchTable`; 4 tests run against a `file://` fake-registry fixture.
- M9.6 — Block + override mechanics (per company marketplace). A company marketplace ships block/override policy *inside its own `catalogue.json`* — there is deliberately no per-user policy file; the rules travel with the marketplace. Two directives, both at qualified-name granularity: `block` lists qualified names (`hex/lodash-helpers`) hidden from discovery and refused by resolution; `override` redirects a bare name to a qualified target (`db-postgres` → `acme/db-postgres`) so bare-name resolution prefers the company's version. The new `core/marketplace/policy.ts` provides `loadAggregatePolicy`, which folds every configured marketplace's policy into one `AggregatePolicy` (blocks union; first marketplace to claim a bare-name override wins; an unreachable registry warns rather than failing). `createAggregateCatalogue` drops blocked entries from `search`/`browse`; `resolveAddress` refuses a blocked qualified address and redirects a bare name through its override (keeping the caller's version spec). The catalogue module's fetch+parse is factored into a shared `fetchCatalogueDoc`. 7 tests cover policy aggregation, block-hides-an-entry (discovery + resolution), and override-prefers-local-version.
- M9.5 — Multi-marketplace aggregation + ordering. `~/.hex/config.yaml` now accepts a `marketplaces:` block — an ordered list of `{ id, registry }` entries (ids validated against the address-qualifier charset, duplicates rejected). Order is significant: it drives bare-name resolution precedence (M9.4's `resolveAddress` walk) and the order discovery results appear in. The new `createAggregateCatalogue` (`core/catalogue/aggregate.ts`) folds every configured marketplace into one flat discovery surface: `search` and `browse` union results across all marketplaces, each entry tagged with its originating marketplace id — so a name clash (the same package in two marketplaces) surfaces as two distinct qualified entries rather than one shadowing the other; `listVersions` returns the first marketplace (declared order) that publishes the package. One unreachable marketplace does not sink discovery — per-marketplace failures land in a `warnings` channel and the rest still answer. `HexConfig.marketplaces` is optional in the type (hand-built partial configs need not spell it out) but always populated by `loadConfig`. 12 tests cover config parsing, the aggregate union, name clashes, ordering, and resilience.
- M9.4 — Qualified-name addressing `<marketplace>/<name>`. The new `core/marketplace/address.ts` parses template addresses in qualified (`hex/api-express@^1.0`) and bare (`api-express@^1.0`, `db-postgres`) forms — `/` separates a marketplace id from the package name and is reserved, `@` introduces the version (defaulting to `latest`), and two-component specs like `^1.0` are normalised to a full triplet. `resolveAddress` resolves an address against an ordered list of configured marketplaces: a qualified address pins exactly one marketplace (an unknown id fails with an error naming the configured ones), while a bare address walks the marketplaces in order — the first whose catalogue lists a version satisfying the spec wins, marketplaces that lack the package are skipped. 9 tests cover the parser, qualified pinning, bare resolution order, fall-through, and the missing-marketplace error. Wiring the marketplace list from `~/.hex/config.yaml` is M9.5.
- M9.3 — `Catalogue` interface (search, browse, list-versions). `idea.md` §9 splits *discovery* from *fetch*: `Source` answers "fetch me this `(name, version)`", `Catalogue` answers "what is out there?". The new `core/catalogue/` module defines the `Catalogue` interface (`search`, `browse`, `listVersions`) and `createMarketplaceCatalogue(registry)` — the marketplace's implementation, reading a registry-wide `catalogue.json` for search/browse and the per-package `index.json` for versions. Search is a case-insensitive substring match across name, description, and categories; browse filters by category; `listVersions` returns published versions newest-first. Only a marketplace gets a `Catalogue` — `FileSource` and `GitSource` have nothing to search, and no-op stubs would fuse abstractions the design keeps apart. 10 tests against a `file://` fake-registry fixture.
- M9.2 — `MarketplaceSource`, the third `Source` implementation. `src/core/marketplace/source.ts` resolves a marketplace coordinate `{ registry, name, version }` into a verified `ComponentBundle`: it loads the registry index (`<registry>/<name>/index.json`), picks the highest published version satisfying the semver spec (`^`, `~`, exact, `*`/`latest` — reusing M6's `versionSatisfies`), downloads the `.hexpkg`, and verifies its signature with M9.1's `verifyPackage` *before* unpacking — an unverifiable package never reaches the cache. Caching mirrors `GitSource` under `~/.hex/cache/marketplace/`: the index and every unpacked package are cached, so a warm cache means zero network, and `refresh` forces a re-fetch. The fetcher dispatches on protocol (`file://` reads from disk, `http(s)://` uses `fetch`) and is injectable for tests. The returned bundle carries `sourceKind: 'git'` so the M7.6 trust gradient always sandboxes marketplace code. 10 tests against a `file://` fake-registry fixture cover semver resolution, verification-gated unpack, and cache reuse.
- M9.1 — Marketplace package format + signing. Defines `hexpkg/1`, the on-the-wire form of a published component or recipe: a gzipped tar archive carrying the authored bundle tree under `artifact/`, a `hexpkg.json` manifest (identity + a per-file sha256 table + an overall digest), and a detached `hexpkg.sig`. Signing is Ed25519 via Node's built-in `node:crypto` (no new crypto dependency); the signature covers a canonical, deterministic serialization of the manifest, so tar mtime/ordering quirks can never destabilise it. The trust model is marketplace-as-signer (the VS Code Marketplace model) — clients pin the marketplace's public key, and a `keyId` field lets the format survive key rotation and multiple marketplaces. The new `src/core/marketplace/package.ts` module provides `packPackage` / `verifyPackage` / `generateSigningKeypair`; verification layers tamper-detection (changed file, added/removed file, substituted digest, forged signature each fail independently). `docs/marketplace-package-format.md` records the format spec and the signing-scheme decision with the comparison to VS Code / npm / apt / Sigstore. Adds `tar` as a dependency.
- M8.7 — Stub recipe integration test. `test/integration/stub-recipe.test.ts` renders a recipe that composes the M8.6 `db-postgres` reference component, once with the slot stubbed and once real, and asserts the file-layout delta: the dev/prod entry points and shared client appear identically in both modes, while `fixtures/seed.sql` (Nunjucks-rendered, `{{ app_name }}` resolved) is the *only* path present in stub mode and absent in real mode. Confirms the M8 contract end-to-end — stub mode is additive and prod-clean, flipping it off leaves nothing behind. Closes the M8 epic.
- M8.6 — `db-postgres` reference stubbable component. `templates/db-postgres/` ships a Postgres data-access component that dogfoods the whole M8 stub surface: a `stub: { engine: pg-mem, fixtures: fixtures }` block, separate prod/dev entry points (`src/index.ts` imports the real `pg` client, `src/index.dev.ts` imports the `pg-mem` stub — both return the same `Db` interface), `pg-mem` isolated in `devDependencies`, a `fixtures/seed.sql` that loads into the in-memory database on connect, and a README documenting the real-vs-stub trade-off. Passes `hex lint` clean and is the fixture the M8.7 integration test renders.
- M8.5 — `hex lint` for the stub prod-clean conventions. A new `hex lint <path>` command checks a stubbable component against the three "no stubs in the production build" conventions from idea.md §6: separate `src/index.ts` (prod) / `src/index.dev.ts` (dev) entry points; the in-process stub engine's npm package declared in `devDependencies` (not `dependencies`); and, for out-of-process engines, a `docker-compose.yml` stub service gated behind `profiles: [dev]`. Each check reports pass / fail / skip; any failure exits non-zero, so the command doubles as a CI / marketplace gate. Real-only components (no `stub:` block) report nothing to lint. A `docs/stubbable-components.md` authoring guide documents the conventions.
- M8.4 — Stub fixtures rendering. A stubbable component's `stub.fixtures` directory is now excluded from the normal render walk unconditionally, and — when the component renders in stub mode — emitted into a fixed `<output>/fixtures/` subtree. Fixture files pass through the same Nunjucks engine as scaffolding files (both path segments and contents), so seed data can reference recipe answers; binary fixtures copy verbatim. `renderRecipe` drives this per child from the recipe's `stub: true` slot decision via the new `RenderOptions.stubEnabled`. A `stub.fixtures` path that escapes the component, is missing, or is not a directory fails with a `RenderError`.
- M8.3 — Out-of-process stub-engine dedup at recipe level. A stub-engine catalogue (`src/core/stub/catalogue.ts`) classifies each engine as `in-process` (pg-mem, msw — package-manager hoisting handles dedup) or `out-of-process` (wiremock — runs in its own container, default port 8080). When a recipe has multiple stubbed children declaring the same out-of-process engine, `collectStubServices` folds them into one shared service. `renderRecipe` exposes the deduped list as a `stub_services` answer (so the recipe-root `docker-compose.yml` template can `{% for %}` over it — emitting exactly one service block per engine) and threads each service's coordinates into `provided.STUB_<ENGINE>_HOST` / `_PORT` so any consuming child reaches the shared instance.
- M8.2 — Recipe-side stub-per-slot wiring. A recipe's `composes:` block accepts a new long object form — `db: { component: <spec>, stub?: true }` — where `component` is any of the existing string wire forms (`<name>@<version>`, `file:`, `git+`); `stub: true` is also accepted on the M6.3 slot form (`{ kind, version, stub: true }`). Every `ChildRef` now carries an optional `stub` flag. At resolve time, `stub: true` against a component that declares no `stub:` block (or against a recipe) is rejected with a `RecipeResolutionError` naming the slot. When a child is stubbed, the well-known `stub_enabled` answer (`true`/`false`) lands in that child's render context, so `include:` rules and Nunjucks `{% if stub_enabled %}` can branch on it.
- M8.1 — Manifest `stub:` block on components. A component can declare `stub: { engine: <id>, fixtures?: <path> }` to advertise that it supports stub mode; absent `stub:` means the component is real-only. `engine` is validated against a closed catalogue (`STUB_ENGINES` — initially `pg-mem`, `msw`, `wiremock`, extensible by editing the one list). `fixtures` is an optional non-empty path (M8.4 will render it into the generated tree). The block is rejected on recipes (component-only, like `provides`/`consumes`/`requires`) and rejects unknown keys.

## [0.7.0] — 2026-05-14

### Added

- M7.7 — `node-ts-hooked` reference component dogfooding the M7 JS-hook surface end-to-end. The bundle's `.hex/manifest.yaml` declares both a `pre_render` JS hook (logs the rendering context — answers, recipe metadata) and a `post_render` JS hook (`name: repository`) with its own `prompts:` block. The post_render hook reads the rendered `package.json` via `project.read`, parses it, optionally splices in a `"repository": "github:owner/name"` field based on the hook-prompt answer (validated, with branch-by-branch `log.info`/`log.warn`), and writes back via `project.write`. The accompanying integration test (`test/integration/node-ts-hooked.test.ts`) exercises the full load → top-level prompts → render → pre_render → walk → post_render path, including the empty-coord, malformed-coord, and `--trust-local` parity branches. Closes M7.
- M7.6 — `hex new --trust-local` for unsandboxed dev-loop hook execution. When passed, JS hooks for FileSource components (local `kind: path` source roots or direct path arguments to `hex new <path>`) run unsandboxed in the host Node process via `Function()`, bypassing QuickJS. Convenient while iterating on your own components. Git/marketplace components (`sourceKind: 'git'`) always sandbox regardless of the flag — the trust gradient is the bundle's, not the user's. Loud warning at startup and per executed hook so the bypass is never silent. To plumb the gradient: `ComponentBundle` and `TemplateEntry` gain a `sourceKind: 'file' | 'git'` marker; `loadFromPath(path, sourceKind?)` accepts the explicit tag; discovery + recipe resolver thread the right value based on each child's `ChildRef` variant.
- M7.5 — Hook-defined prompts. A JS hook can declare its own `prompts:` block alongside `js:` (`when:` already supported); shorthand prompt forms are desugared the same way as top-level prompts. The hook also accepts an optional `name:` field — the namespace key under which prompt answers land in `answers.hooks.<name>.*`. Without an explicit `name:`, the filename minus `.js` is used. Prompts fire at the hook's lifecycle moment, just before the hook body runs; answers thread through to subsequent phases (walk, declarative post_render, post_render JS hooks all see the augmented tree). Hooks read the full answers tree (shared read) but persist writes only to their own namespace (isolated write). `renderBundle`/`renderRecipe`/`executeNewRender` gained a `prompter` option, and the CLI threads its clack prompter through both prompt collection and the render call.
- M7.4 — JS hook lifecycle wiring. `renderBundle` now fires `pre_render` JS hooks after the output-dir writeable check + mkdir, before the template walk, so a hook can short-circuit the render by throwing (HookExecutionError aborts before any file is written). `post_render` JS hooks fire after the declarative rename/delete pass, so they observe the final tree shape. Each hook executes inside a freshly-created QuickJS sandbox with four context surfaces installed as globals: `answers` (full prompt tree), `recipe` (`{ name, version }` of the outermost recipe when rendering inside one, `null` for standalone components), `project.*` (sandboxed FS facade from M7.2), and `log.{info,warn,error}` routed to console by default (overridable via `RenderOptions.hookLog`). `JsHook.when` is honoured. Existing declarative + JS hooks coexist on the same lifecycle.
- M7.3 — JS hook discovery + manifest declaration. Component manifests can now declare JS hooks alongside the existing declarative entries: `hooks.pre_render: [{ js: '<filename>' }]` and `hooks.post_render: [..., { js: '<filename>' }, ...]`. The filename refers to a file inside the bundle's `.hex/hooks/` directory (plain `.js` filename — no path separators, no `..`), and the bundle loader eagerly reads the file into `ComponentBundle.jsHookSources` at load time. Missing hook files fail loading with an authoring error naming the lifecycle, filename, and expected path. The existing declarative rename/delete hooks are untouched. JS hooks are discovered now; the sandbox doesn't execute them yet — that lands with M7.4.
- M7.2 — Sandboxed project FS facade. `src/core/hooks/project-fs.ts` exposes `read` / `write` / `delete` / `exists` / `list` over a project root, with a traversal guard that rejects absolute paths, `..` escapes, and symlinks whose realpath resolves outside the root (works for non-existent targets too, by realpath-ing the longest existing ancestor). `Sandbox.installProjectFs(fs)` bridges the five operations into the QuickJS context as `project.*` host functions — rejections surface inside the hook as catchable JS errors with usable messages, not silent failures. Not wired into the render pipeline yet (M7.4).
- M7.1 — QuickJS-WASM hook sandbox foundation. `src/core/hooks/sandbox.ts` embeds the `quickjs-emscripten` runtime (release-sync variant, ~684 KiB WASM blob bundled into `node_modules`, no native binaries, no runtime download). `createSandbox()` returns a fresh isolated runtime + context per call; `runScript()` executes user code with a per-call CPU deadline (default 5 s) and a runtime-wide memory ceiling (default 32 MiB), throwing `SandboxError` on script error, timeout, or OOM. Hooks running here cannot see `require`, `process`, or any Node primitive — only what later M7 work bridges in. Not wired into the render pipeline yet (that lands with M7.4).
- M4 — post-scaffold setup tasks. Templates can declare a `setup:` block in `.hex/manifest.yaml` with an optional `message` and `tasks` (each with `id`, `title`, optional `detail`). Hex tracks user progress in the generated app's `.hex/checklist.yaml`. When `hex new` finishes rendering, it writes the checklist, prints the message, and on a TTY walks the user through each task interactively — `Mark as done` / `Skip for now` / `Quit`. State persists after every toggle so a hard exit cannot lose progress.
- `hex setup` — resume the interactive loop from any directory inside a generated app (walks upward from cwd to find `.hex/checklist.yaml`, matching how `git`/`npm` find their roots). Walks all tasks regardless of status; done tasks can be flipped back to pending.
- `hex doctor` — when run inside a generated app, appends an "Outstanding setup tasks" section listing pending task ids + titles. Stays silent when no checklist is found or every task is already done.
- `hex new --no-setup` — skip the post-render interactive loop. Non-TTY invocations skip automatically and print "N setup tasks pending — run hex setup".
- `node-ts-cli` template gains an `include_publish_workflow` prompt (default true) gating a `.github/workflows/publish.yml` that publishes on `v*` tag, plus a four-task `setup:` block (install deps, init git + push, set NPM_TOKEN, tag first release).
