## [Unreleased]

### Changed

- Lockfile ŌĆö capture the full nested-recipe tree. The M10.1 `LockChild` schema and the M10.2 `buildLockfile` builder now record composed children **recursively**: a recipe child carries its own descendants under a nested `children` field (a component leaf omits it). This closes the M10.2 known limitation ŌĆö only the root's immediate children were captured ŌĆö so the lockfile describes the whole composition tree the M11 upgrade engine needs for pristine reconstruction. The schema's `lockChildSchema` becomes a `z.lazy` recursive type; the builder walks each child's resolved sub-recipe. 3 new tests cover nested-tree schema validation (including recursive rejection of a malformed grandchild) and a three-level `recipe ŌåÆ recipe ŌåÆ component` render whose lockfile records the tree.
- Added a `prepublishOnly` script (`npm run check && npm run build`) so any future `npm publish` is forced to run the full test suite and recompile `dist/` from current source first ŌĆö the published tarball can never be a stale build, and a failing test aborts the release. Purely a release-time guard; no behaviour change day-to-day. The package is not yet published.

### Changed

- Manual-test matrix docs ŌĆö three release-time test surfaces don't lend themselves to cheap CI: the real `@clack/prompts` wiring against a TTY (M4 known gap), non-`file://` git auth paths (HTTPS public, HTTPS credential-helper, SSH agent ŌĆö M3 known gap), and Windows shell-to-git (M3 known gap, all current CI runs on darwin). Each of those tickets' own acceptance criteria allowed a documented manual-test matrix as the alternative to building heavy infra (pty harness, containerised git server, Windows CI matrix). New `docs/testing.md` is that matrix ŌĆö section per surface, **specific commands** to run and **specific outputs** to look for, plus a release checklist. README's `## Scripts` section now points at it. Closes M4 clack-prompter end-to-end (`869d6q7wx`), M3 real auth paths (`869d6q7wn`), M3 Windows shell-to-git (`869d6q7wh`) ŌĆö the last three items in the tech-debt sweep that followed the M11 epic.

### Fixed

- Action-handler coverage thin ŌĆö the commander `.action()` callbacks for `hex new` (post-render TTY-vs-`--no-setup` decision) and `hex setup` (no-checklist-found branch with its stderr message + `process.exitCode = 1`) were only covered by indirection. Fixed by mirroring the M3.7 pattern: pull the branch logic out of the callback into pure / dependency-injected functions. `src/commands/new.ts` gains a pure `planPostRender(result, { isTTY, setup })` returning `{ kind: 'no-tasks' } | { kind: 'has-tasks', interactive, initial, pendingCount, setupMessage? }` ŌĆö the `isTTY && opts.setup` decision is now one expression in a function, not a 2-line `if` in the spinner block. `src/commands/setup.ts` gains a `runSetupCommand(cwd, effects)` taking a `SetupCommandEffects` (stderr / setExitCode / printIntro / printOutro / prompterFactory), plus a `defaultSetupCommandEffects` the production registrar wires; tests inject capturing effects to assert the no-checklist branch sets exit code 1 and writes the user-facing message to stderr without ever touching `process`. 7 new tests ŌĆö 5 for `planPostRender` (no-tasks, every isTTY ├Ś setup combination, setupMessage propagation) and 2 for `runSetupCommand` (the no-checklist branch + the happy-path session drive).
- Checklist concurrency ŌĆö two `hex setup` invocations against the same generated app's `.hex/checklist.yaml` (or one running while `hex new` finalises) could clobber each other's toggles via last-writer-wins on the whole file. The setup loop now passes the *specific* toggle (`{ taskId, status }`) to its `onSave` callback, and a new `updateChecklist(rootDir, mutate)` helper applies the mutator inside the same `withFileLock` primitive M3 cache concurrency added: it serialises read-modify-write, re-reads the latest disk state inside the lock, and writes back atomically ŌĆö so two terminals each toggling a different task end up with *both* toggles on disk, not one. `writeChecklist` keeps its existing role (full overwrite, e.g. the initial write by `hex new`) but now writes atomically via `writeFileAtomic`, so a concurrent reader never sees a half-written file. `hex setup`'s persist path writes through `writeChecklist` if no file exists yet (first-toggle creation) and through `updateChecklist` otherwise. 4 unit tests cover the new helper (mutator applied + reloads clean, throws when no checklist, rejects a mutator that produces a malformed shape, and a 5-worker concurrent-toggle race where every toggle lands).
- Cache concurrency ŌĆö two `hex` invocations against the same `(url, ref)` could race during clone/refresh: the meta file write (`.hex-meta.json`) wasn't atomic, and the whole read-then-mutate flow in `resolveGitSource` had no mutual exclusion, so two cold resolves could both try to `git init` / `git fetch` into the same repo dir and a `checkUpstreamDrift` could stomp the freshly-written meta. A new `src/core/util/atomic.ts` ships two dependency-free primitives ŌĆö `writeFileAtomic` (write-to-temp + `rename`, so a reader observes either previous or new bytes, never a half-written file) and `withFileLock` (exclusive-create lockfile with `staleMs` reclaim, so a crashed process doesn't wedge things). `writeMeta` becomes atomic; `resolveGitSource` keeps the warm-cache fast path lock-free (safe now that reads are consistent) and serialises the clone/refetch under a `.lock` sentinel in the cache root, re-reading meta inside the lock so a peer-completed refresh short-circuits; `checkUpstreamDrift` runs `ls-remote` outside the lock (it's a slow network call) and persists the result under the lock so a concurrent resolve can't lose `lastCheckedAt` / `lastKnownUpstreamSha`. 8 tests cover the primitives (atomic write, lock serialisation, release on success/throw, stale reclaim, timeout, parent-dir requirement) and 2 cover end-to-end concurrency (5 cold resolves and 5 warm-cache refreshes against the same upstream ŌĆö all converge on the same sha with a valid meta and no leftover lock).

### Added

- M11.9 ŌĆö Documentation: upgrade flow + migration authoring guides. Closes the M11 epic. `docs/upgrade.md` is the user-facing workflow for `hex upgrade` ŌĆö what the command does in plain language (the snapshot, the pristine-tree reconstruction, the chain walk, the 3-way merge, the lockfile rewrite), the conflict resolution flow with marker syntax and `--continue` / `--abort` semantics, orphan handling and the opt-in `--prompt-on-orphans` flag, an exit-code table, and a troubleshooting list. `docs/authoring-migrations.md` is the format + authoring guide for template maintainers ŌĆö file layout (`.hex/migrations/<from>-to-<to>.{yaml,js}`, one per hop, composed across multi-version upgrades), the four declarative verbs (`rename`, `replace`, `delete`, `delete_if_unmodified`) with a table of which tree each op routes to, the JS escape hatch's sandbox surface (`project` / `log` / `migration`), the user-tree escape hatch (`user_tree: true` for declarative, `.user.js` for JS), authoring rules, and a pointer to the integration test as the canonical dogfood pattern. The README is updated: the stale "Phase 1 ŌĆö M1ŌĆōM3" status line now reflects M1ŌĆōM11 shipped, a new "Upgrading a generated app" subsection links both docs, and the Roadmap blurb is current.
- M11.8 ŌĆö End-to-end upgrade integration test + composed migration chain. `test/integration/upgrade.test.ts` dogfoods the whole M11.1ŌĆōM11.7 stack against real rendering ŌĆö staging a fixture component template across three versions, rendering v1.0.0 into a user app (with a real lockfile), editing it, and running `runUpgrade` to v2.0.0. Writing it surfaced a real gap: `runUpgrade` applied only the *last* hop's migration to `pristine_new`, so intermediate-hop migrations in a multi-version chain ran against a discarded tree and never affected the outcome. **Fixed:** the chain walk now only *discovers* every hop's migration; `runUpgrade` then applies the whole **composed chain**, in hop order, before the merge ŌĆö so every migration is exercised. The application is also corrected to be structurally sound: a `rename` / `replace` realigns the file in `pristine_old` **and** the user's working tree (so a user's edit follows the template's rename and 3-way merges into the renamed file rather than orphaning), while a `delete` / `delete_if_unmodified` prunes `pristine_new` (so the merge ŌĆö and M11.7's orphan handling ŌĆö treats it as a template removal). JS migrations transform `pristine_old`. This makes the path-keyed 3-way merge rename-aware for free, with no change to `merge.ts`. Two integration tests: the **clean** path ŌĆö edits to a plain file *and* to the file v1.1.0 renames both survive, folded with the template's own changes; the `delete`/`replace` migrations land; the lockfile bumps to v2.0.0 and integrity verifies clean ŌĆö and the **conflict** path ŌĆö a clashing edit stops with markers + persisted `.hex/upgrade-state.yaml`, then `continueUpgrade` finishes once resolved. Plus 2 unit tests in `run.test.ts` for a rename carrying an edit and a delete pruning a file.
- M11.7 ŌĆö Modify/delete + orphan handling defaults. When a template upgrade removes a file the user has *edited*, deleting it would throw away their work ŌĆö so the engine **preserves and orphans** it (`idea.md` ┬¦1, "preserve and orphan"). M11.4's merge already kept such files; M11.7 formalises the contract. `mergeTrees` now reports an `orphaned` list in `MergeResult` (an untouched removed file is still deleted; only edited ones orphan), and accepts an optional `onOrphan` triage callback ŌĆö absent, every orphan is kept. `runUpgrade` threads the orphans through: on a clean upgrade `finalizeLockfile` records them under a new optional `orphans` field on the lockfile schema (so `hex doctor` and a later upgrade can tell kept user orphans from template-owned files); on the conflict path they ride in `.hex/upgrade-state.yaml` under a new `orphans` field, and `continueUpgrade` writes them to the lockfile once conflicts resolve. `UpgradeOutcome` gains `orphans` on both variants. The `hex upgrade` command prints an end-of-run `ŌÜĀ N orphaned file(s) have your edits ŌĆö kept in place` summary listing the paths, and a new opt-in `--prompt-on-orphans` flag triages each orphan interactively (keep / delete) via a `@clack/prompts` selector ŌĆö a cancel keeps the file. 5 tests cover the merge layer (orphan reported, `onOrphan` electing delete) and the full upgrade (edited-removed file preserved + recorded in the lockfile, and the delete decision dropping it).
- M11.6 ŌĆö User-tree migration escape hatch. Some migrations genuinely need to touch user code ŌĆö a project-wide import rename ŌĆö which the pristine-tree model can't express. A migration may now declare itself a **user-tree migration**: it bypasses the pristine tree and runs against the user's working copy instead. A declarative migration opts in with `user_tree: true` (a new optional field on `migrationDocSchema`); a `.js` migration opts in via the `.user.js` extension (a bare `.js` file carries no schema to hang a flag on ŌĆö `discoverMigration` rejects a hop that ships both `.js` and `.user.js`). `discoverMigration` now reports a `userTree` flag and carries the parsed declarative `doc`, and a new `runDiscoveredMigration` runs an already-discovered migration so the engine never re-reads it. In `runUpgrade`, the chain walk collects user-tree migrations instead of applying them to the pristine tree; they run **last**, against the freshly merged working copy, via `applyUserTreeMigrations`, which snapshots the tree before and after so the reported `userTreeChanges` is exactly the escape hatch's footprint. The CLI prints a `ŌÜĀ a user-tree migration edited your code directly` notice listing those files; on the conflict path they are also recorded in `.hex/upgrade-state.yaml` under a new `user_tree_changes` field. `UpgradeOutcome` gains `userTreeChanges` on both variants. 6 tests cover discovery (`user_tree`-declarative, `.user.js`, the both-JS-forms rejection, pristine default) and the full upgrade ŌĆö a `.user.js` codemod renaming an import across `src/` files routed against the working copy on the clean path, and the change recorded in upgrade-state on the conflict path.
- M11.5 ŌĆö `hex upgrade` command + `--continue` flow. `core/upgrade/run.ts` ties M11.1ŌĆōM11.4 into the upgrade flow, and a new `hex upgrade` command exposes it. `runUpgrade` snapshots the working tree (so `--abort` can roll back exactly), walks the version chain rendering + migrating each hop, and 3-way merges the result onto the user's tree: a **clean** merge rewrites the lockfile at the new version (re-hashing the merged tree via the now-exported `hashTree`) and finishes; a **conflicting** one leaves marker-laden files in place and persists `.hex/upgrade-state.yaml`. `continueUpgrade` finalises a paused upgrade ŌĆö refusing while any conflicted file still carries `<<<<<<<` / `>>>>>>>` markers ŌĆö then rewrites the lockfile and clears the state + snapshot. `abortUpgrade` restores the working tree from the snapshot and clears everything. *Where versions come from* and *how a version is rendered* arrive as an injected `UpgradeEnvironment`, so the orchestration stays agnostic to file / git / marketplace sourcing. The `hex upgrade <new-template>` command wires a concrete environment: `pristine_old` from the lockfile (M11.1), `pristine_new` from the supplied newer template; `hex upgrade --continue` / `--abort` finish or discard a paused run. A clean upgrade exits 0; a conflicting one prints the conflicted files and exits non-zero. 8 tests cover all four paths ŌĆö clean merge + lockfile rewrite, conflict + state persistence, `--continue` (resolved and still-unresolved), and `--abort` rollback ŌĆö plus the in-progress and no-upgrade guards.
- M11.4 ŌĆö 3-way merge with git-style conflict markers. `core/upgrade/merge.ts` adds the engine that replays a template change onto the user's working tree (`idea.md` ┬¦1, "conflict UX, git-style markers"). `threeWayMerge` does a single file's 3-way merge ŌĆö base / ours / theirs ŌĆö via the new `node-diff3` dependency, returning git's *default*-style `<<<<<<< ======= >>>>>>>` markers (no `|||||||` base section) so every editor and `git diff` reads them. `mergeTrees` diffs `pristine_old` against `pristine_new` and applies the change file by file: a template-changed file is 3-way merged against the user's copy (landing cleanly when the user never touched it); a template-added file is written, or merged against an empty base when the user already created one there; a template-removed file is deleted when untouched and kept as an orphan when the user edited it (M11.7 formalises the orphan warning). `.hex/`, `.git/`, and `node_modules/` are never touched. Conflicted files are written **in place** with markers and listed in the result, so the user resolves them exactly as a rebase. `core/upgrade/state.ts` adds the `.hex/upgrade-state.yaml` file ŌĆö a zod-validated record of an in-flight upgrade (`from`, `to`, the conflicted paths) that `hex upgrade --continue` will read and `--abort` will clear; it exists only while an upgrade is paused. 24 tests cover the single-file merge, every tree-merge path (clean add/update/delete, user-edit preservation, real 3-way merge, conflict markers, orphan retention, `.hex/` exclusion), and the state file's write/read/clear lifecycle. Adds `node-diff3` as a dependency.
- M11.3 ŌĆö Migration script vocabulary. `core/upgrade/migration.ts` adds the per-hop migration the chain walker (M11.2) runs. A version bump can ship a migration at `.hex/migrations/<from>-to-<to>.{yaml,js}`; `discoverMigration` finds it (a hop with no migration is a clean no-op, and a hop with *both* forms is an authoring error). The declarative YAML form is an ordered `steps:` list over four verbs ŌĆö `delete` (unconditional), `delete_if_unmodified` (kept when the user has edited it, via an injected `isUserModified` predicate), `rename` (carry a path forward), and `replace` (rename flagged as a content change). Every op transforms a *pristine* tree, never the user's working copy, so the upgrade diff stays deterministic. The `.js` form is the escape hatch: it runs inside the M7 sandbox with the same `project` filesystem facade JS hooks get (scoped to the pristine tree), plus a `log` sink and a `migration` global naming the hop. `applyMigration` returns the structural changes it made ŌĆö `renames` (each tagged `rename`/`replace`) and `deletes` ŌĆö so the M11.4 3-way merge can build a rename-aware diff. 13 tests cover discovery (no-op, both-forms rejection), every declarative op (including order, the modified/unmodified branches, and source-missing/target-exists errors), schema rejection, and the JS escape hatch (a sandboxed migration mutating the tree, and a throwing one surfacing as a `MigrationError`).
- M11.2 ŌĆö Stepwise migration chain walker. `core/upgrade/chain.ts` adds the walker that drives a multi-version upgrade one hop at a time (`idea.md` ┬¦1, "stepwise chain") ŌĆö an app on v1 upgrading to v4 walks v1ŌåÆv2ŌåÆv3ŌåÆv4, so a template author never writes a v1ŌåÆv4 migration. `buildVersionChain(from, to, available)` is the pure semver core: it sorts the published versions, includes both ends, walks only versions that were actually published (gaps allowed), and throws when either end is unpublished or the target is not strictly newer. `walkUpgradeChain` orchestrates the hops: it renders each version's pristine tree and runs that hop's migration against it, threading the prior step's tree through as `fromTree` so a migration can carry user edits forward. *How* a version is rendered and *how* a migration is applied arrive as injected callbacks ŌĆö rendering a version generalises M11.1's pristine reconstruction, and applying a migration is M11.3; wiring the real implementations is the `hex upgrade` command (M11.5). The walker only ever writes to the temp directories `renderVersion` hands back ŌĆö never the user's tree ŌĆö and on any mid-chain failure removes every directory it created and rethrows, so an aborted upgrade leaves nothing behind; on success it keeps `pristine_old` + `pristine_new` and cleans the intermediate step trees. 10 tests cover the chain arithmetic and the walk (hop order, tree threading, intermediate cleanup, failure rollback).
- M11.1 ŌĆö Pristine reconstruction from the lockfile. Opens the M11 ŌĆö Upgrade engine epic. The new `core/upgrade/` module's `reconstructPristine(lockfile, opts)` rebuilds `pristine_old` ŌĆö the rendered tree exactly as it looked when the user first ran `hex new` ŌĆö and returns the path to it (a fresh `mkdtemp` directory, or a caller-supplied `targetDir`). It re-fetches every locked artifact from its recorded `source` spec (`file` and `git` today; a `marketplace` source fails loudly, since `hex new`'s resolver does not yet produce one), then replays the stored `answers` through the render engine. Crucially it does **not** re-resolve the recipe's `composes:` block ŌĆö that would pick up whatever versions discovery sees now; instead it rebuilds the `ResolvedRecipe` directly from the lockfile's recorded child tree (recursing into nested recipes), so the output reproduces the historical render byte-for-byte regardless of what has since been published. Both archetypes go through one path ŌĆö a standalone component renders via `renderBundle`, a recipe via `renderRecipe` ŌĆö with no `if (recipe)` branch past the single dispatch. Because the render engine is deterministic and hooks (declarative + JS) re-run identically, `checkLockfileIntegrity` against a reconstructed tree returns clean. 4 tests prove it: a recipe round-trip whose `api` child carries a post_render rename hook (so reconstruction must replay the hook to verify clean), a standalone component, the `mkdtemp` default, and the marketplace-source rejection.
- M10.5 ŌĆö Tests + dogfood: lockfile written for the M5 reference recipe. Closes the M10 ŌĆö Lockfile epic by exercising the lockfile end-to-end against the existing integration tests. The M5 recipe pipeline test (`test/integration/recipe.test.ts`) now writes a lockfile after rendering the `demo-app` reference recipe and asserts it reloads through the schema, that its `root` + the two children (`api`, `web`) carry the right identities/versions/stub flags, that `answers` equals what the render consumed, and that `checkLockfileIntegrity` returns clean on the freshly-rendered tree. The M8.7 stub integration test (`test/integration/stub-recipe.test.ts`) dogfoods the stub surface: rendering the `db-postgres` reference component slot stubbed *and* unstubbed, it confirms the lockfile records each child's `stub` flag faithfully and that integrity verifies clean either way ŌĆö proving the M8 stub decision round-trips into the M10 lockfile. This is the loop `idea.md` ┬¦11 calls for between the render milestones and the M11-enabling lockfile.
- M10.4 ŌĆö `hex doctor` integration: lockfile health. When `hex doctor` finds a `.hex/lockfile.yaml` in the cwd or an ancestor (M10.3's `readLockfileUpward`), it now appends a "Lockfile" section ŌĆö mirroring how the M4 checklist's "Outstanding setup tasks" section already appears. The block prints the root recipe/component identity (`name@version`), the composed-children tree with each child's version, recursed into nested recipes with deeper indentation, and stub children flagged. Below it, a single integrity line runs `checkLockfileIntegrity` against the current tree: a green `Ō£ō integrity clean`, or a `ŌÜĀ N files diverged from the lockfile` with a `modified / missing / added` breakdown. An absent lockfile leaves doctor silent (same as the setup section); a *present but unreadable* one ŌĆö malformed, or written by a newer Hex ŌĆö surfaces as a one-line warning instead of the full section. The renderer (`formatLockfileSection`) is pure and exported; 8 tests cover the clean and divergent states, nested children, the singular/plural divergence wording, the not-checked case, and a standalone component.
- M10.3 ŌĆö Read the lockfile + integrity verification. `core/lockfile/` gains the read side: `readLockfileUpward(startDir)` walks upward from any directory looking for `.hex/lockfile.yaml` ŌĆö the same convention `readChecklistUpward` uses, so `hex doctor` (M10.4) and the M11 upgrade engine work from any subdirectory of a generated app; it returns the loaded lockfile with its path + app root, or `null`. The reader checks `schema_version` **before** full schema validation: a file written by a newer Hex is refused with an explicit "upgrade Hex" hint rather than a wall of schema errors from shapes a future format introduced; malformed YAML or a schema mismatch raises a `LockfileError`. `checkLockfileIntegrity(rootDir, lockfile)` compares the current tree against the recorded hash table and **never throws** ŌĆö it returns `{ ok, modified, missing, added }` so the caller (M11's pristine reconstruction, doctor) decides whether to merge, warn, or proceed: `modified` is recorded files whose bytes changed, `missing` is recorded files now gone, `added` is files new since generation. The comparison walk excludes `.hex/`/`.git/`/`node_modules/`, identical to the `buildLockfile` walk. 10 tests cover the happy path, upward walk from a nested subdir, the no-lockfile case, future-version rejection, structural-invalid rejection, and each integrity divergence (modified / missing / added).
- M10.2 ŌĆö Write the lockfile at the end of `hex new`. After a successful render, `hex new` now writes `.hex/lockfile.yaml` into the generated app ŌĆö alongside the M4 checklist, in the same `.hex/` folder ŌĆö so the app becomes self-describing. `core/lockfile/` gains `buildLockfile` + `writeLockfile`: the builder records the root artifact, the recipe's immediate composed children (each with its `composes:` slot key + stub flag), the full answers tree, and a per-file sha256 table; the writer validates against the M10.1 schema before persisting, so a buggy caller cannot emit a malformed file. The hash table is computed by walking the rendered tree **on disk** ŌĆö post-hooks, post-render ŌĆö so hook renames/deletes are reflected faithfully; `.hex/`, `.git/`, and `node_modules/` are excluded (the first is Hex's own metadata, hashing it would make the table self-referential). Both archetypes go through the one path ŌĆö `executeNewRender` writes the lockfile with no `if (recipe)` branch. Source specs are captured as precisely as the pipeline exposes: a `git:` child reference keeps its upstream coordinate verbatim, while `file:`/`name`/`slot` references and the root are recorded as their resolved local path. Known limitation: only a recipe's immediate children are captured ŌĆö nested-recipe descendants are deferred to the M11 upgrade work, which needs the full tree. 2 tests assert a schema-valid lockfile whose hashes match the bytes on disk, for a recipe (with children) and a standalone component (none).
- M10.1 ŌĆö Lockfile schema. The first M10 ticket defines `.hex/lockfile.yaml`, the file that makes a generated app *self-describing* ŌĆö the inverse of an authored component's `.hex/manifest.yaml` (manifest: *how to scaffold*; lockfile: *what was scaffolded*). The new `core/lockfile/` module ships a zod schema for the document: a `root` artifact (the recipe ŌĆö or, with no `children`, a standalone component, so the engine keeps its no-`if (recipe)` uniformity), a `children` array (each with its `composes:` slot key + stub flag), the full `answers` tree, and a per-file `files` table of lowercase-hex sha256 digests over the rendered tree. Each artifact carries a discriminated `source` spec ŌĆö `file` / `git` / `marketplace`, one variant per `Source` implementation ŌĆö so M11 can re-fetch the exact bytes for pristine reconstruction. `schema_version` is `1`; the base schema accepts any positive integer there on purpose, leaving a future-version file for the M10.3 reader to refuse with an upgrade hint rather than a bare schema error. sha256 reuses the marketplace package format's hashing primitive ŌĆö one content-hash algorithm across the codebase. `docs/lockfile-format.md` is the format spec; 9 tests cover the shape, every source-spec variant, and tampered/malformed-hash rejection. Writing the file is M10.2; reading + integrity verification is M10.3.
- M9.9 ŌĆö Public Hex marketplace MVP (registry server). A new top-level `registry/` holds the marketplace registry server ŌĆö a single dependency-free Node `http` service that closes the M9 epic by hosting the package format the rest of the epic was built against. **Read side**: `GET /catalogue.json`, `GET /<name>/index.json`, `GET /packages/<file>.hexpkg` ŌĆö the exact static-shaped endpoints `MarketplaceSource` (M9.2) and the `Catalogue` (M9.3) fetch. **Publish**: `POST /publish` accepts a bearer-token-authenticated upload of an *unsigned* bundle tarball, validates it, **signs it with the marketplace key**, and ingests it ŌĆö developers hold a publish token, not a signing key (marketplace-as-signer, M9.1); publishes are serialised so catalogue/index regeneration never interleaves, and an existing `name@version` is refused (immutable). **Website**: server-rendered search / browse / per-package detail pages, progressively enhanced with HTMX (the `htmx.org` dependency, served from `/assets/htmx.min.js` ŌĆö never a third-party CDN) for live search. A new `hex publish <dir>` CLI command packs a bundle and uploads it; `registry/keygen.ts` (`npm run registry:keygen`) generates the marketplace keypair. `docs/registry-deploy.md` is the deploy runbook (keypair, tokens, env, TLS, smoke test). 13 tests cover the store, publish flow, every endpoint, and an in-process publishŌåÆfetch loop that resolves a just-published package back through `MarketplaceSource` with signature verification. DNS / TLS / hosting on Textology infrastructure remain for the operator.
- M9.8 ŌĆö `hex browse` command. The sibling discovery surface to `hex search`: `hex browse` lists the marketplace-curated categories across every configured marketplace (a `name (count)` tally folded from the aggregate `Catalogue`), and on a TTY presents an interactive picker that drills down into the chosen category's entries. `hex browse <category>` lists a category directly, skipping the picker; piped (non-TTY) output prints the flat category list. `--json` emits `{ categories, warnings }` (or `{ category, results, warnings }` for a category argument). Block policy is honoured ŌĆö blocked entries never reach the category tally or the drill-down. Drill-down entries reuse `hex search`'s table formatter. The testable core is `listCategories` + `browseCategory` + `formatCategories`; 6 tests run against a `file://` fake-registry fixture.
- M9.7 ŌĆö `hex search` command. The first CLI surface for the marketplace: `hex search <query>` walks every marketplace in `~/.hex/config.yaml`, aggregates the hits via the M9.5 aggregate `Catalogue`, and prints aligned `qualified-name@version  type  description` rows. Block/override policy is loaded and applied ŌĆö blocked entries are dropped from results. `--json` emits `{ results, warnings }` for scripting (mirroring `hex list` / `hex sources`); unreachable marketplaces and policy-load failures surface as warnings rather than aborting.
